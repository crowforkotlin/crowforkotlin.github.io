Arch Linux UKI 模式下 GRUB 启动项异常排查与修复

发表于2026-07-15|更新于2026-07-17|Unix

一、背景知识

在排查问题之前，需要先了解 GRUB 启动流程中涉及的几个关键组件。

1.1 GRUB（GRand Unified Bootloader）

GRUB 是 Linux 系统常用的引导加载程序。计算机启动时，UEFI 固件首先加载 GRUB，GRUB 再根据配置文件（grub.cfg）将操作系统内核加载到内存中。

GRUB 的配置文件由 grub-mkconfig 命令生成，它会自动扫描系统中的 /etc/grub.d/ 目录，按文件名中的数字前缀顺序依次执行其中的脚本，将所有脚本的输出合并为最终的 grub.cfg。

重要规则：grub-mkconfig 仅执行具有可执行权限（x）的脚本。通过 chmod -x 可以禁用某个脚本，使其不参与配置生成，而无需删除该文件。

1.2 10_linux 脚本

10_linux 是 GRUB 自带的 Linux 内核检测脚本。它的工作方式是传统模式：

  • 在 /boot 目录下查找内核文件（vmlinuz-linux）
  • 查找初始化内存盘文件（initramfs-linux.img）
  • 查找 CPU 微码文件（intel-ucode.img 或 amd-ucode.img）

这种方式要求内核、initramfs、微码作为三个独立文件分别加载。

1.3 15_uki 脚本

15_uki 是 GRUB 2.12 版本新增的脚本，用于支持 UKI 启动模式。它的工作方式与 10_linux 完全不同：

  • 不会在 grub-mkconfig 执行时生成具体的启动项
  • 仅在 grub.cfg 中写入一条 uki 命令
  • GRUB 在实际启动时执行该命令，动态扫描 EFI 分区中的 UKI 文件并生成启动项

1.4 UKI（Unified Kernel Image，统一内核镜像）

UKI 是一种将多个启动组件打包为单个 EFI 可执行文件的技术。一个 UKI 文件包含：

组件 说明
Linux 内核 操作系统核心
initramfs 初始化内存盘，包含启动早期所需的驱动和工具
CPU 微码 Intel/AMD 处理器的微码更新
内核命令行参数 如 root=、quiet 等启动参数

1.5 40_custom 脚本

40_custom 是 GRUB 提供的用户自定义启动项文件。用户可以在其中手动编写 menuentry，这些自定义条目会在 grub-mkconfig 生成配置时被原样包含。

由于 40_custom 的内容不会被自动覆盖或修改，它是实现持久化自定义启动项的推荐方式。

1.6 传统模式与 UKI 模式的对比

对比项 传统模式（10_linux） UKI 模式
文件结构 内核、initramfs、微码分开存放 打包为单个 .efi 文件
GRUB 加载方式 linux + initrd 分步加载 chainloader 一步加载
生成工具 grub-mkconfig 自动检测 mkinitcpio 生成，15_uki 检测
配置文件 /etc/grub.d/10_linux /etc/grub.d/15_uki
mkinitcpio preset default_image="/boot/initramfs-linux.img" default_uki="/boot/EFI/Linux/arch-linux.efi"

2.1 分区布局

三、问题现象

3.1 初始状态

安装 GRUB 并执行 grub-mkconfig 后，重启进入 GRUB 菜单，出现以下 5 个启动选项：

序号 选项名称 图标 启动结果
1 Arch Linux 有（BigSur 主题 arch 图标） 蓝屏（内核恐慌）
2 Arch Linux 高级选项 — 子菜单
3 Arch Linux（UKI 动态生成） 无 正常进入系统
4 Windows Boot Manager 有 正常
5 UEFI Firmware Settings — 正常

3.2 问题表现

  • 第 1 个选项（有图标）：选择后出现蓝屏/内核恐慌，系统无法启动
  • 第 3 个选项（无图标）：可以正常启动并进入 Arch Linux 桌面
  • 两个 Arch Linux 选项共存，用户需要手动选择第二个才能正常启动

四、原因分析

4.1 根本原因：10_linux 在 UKI 模式下生成了不完整的启动项

10_linux 脚本使用传统方式生成启动项，要求系统中存在独立的 initramfs-linux.img 文件。但在 UKI 模式下，mkinitcpio 的 preset 配置可能仅生成 UKI 文件，不生成独立的 initramfs-linux.img。

10_linux 在扫描 /boot 目录时的行为：

  1. 找到 vmlinuz-linux → 写入 linux /vmlinuz-linux
  2. 找到 intel-ucode.img → 写入 initrd /intel-ucode.img
  3. 未找到 initramfs-linux.img → 跳过，不写入

4.2 缺少 initramfs 的后果

initramfs（初始化内存盘）包含启动早期阶段所需的驱动程序和工具。内核加载后需要通过 initramfs 来挂载根文件系统。缺少 initramfs 时，内核无法访问磁盘控制器驱动或文件系统驱动，导致根文件系统挂载失败，触发内核恐慌（Kernel Panic），表现为蓝屏。

4.3 两个脚本的冲突

10_linux 和 15_uki 各自独立运行，互不感知：

两个脚本不知道对方的存在，也无法判断用户实际使用的是传统模式还是 UKI 模式，因此同时生成了两种启动项。

4.4 为什么蓝屏的选项有图标，正常的反而没有

GRUB 主题通过 menuentry 中的 --class 参数匹配图标。10_linux 生成的条目包含 --class arch，BigSur 主题中存在对应的 arch 图标文件，因此显示图标。

15_uki 在启动时动态生成的条目，其图标匹配逻辑与静态条目不同，可能未正确关联到主题的图标文件，因此不显示图标。

五、错误修复尝试

在排查过程中，以下操作导致了更严重的问题：

5.1 错误操作

  1. 禁用 15_uki 脚本：sudo chmod -x /etc/grub.d/15_uki
  2. 删除 UKI 文件：sudo rm /boot/EFI/Linux/arch-linux.efi

5.2 错误原因

上述操作基于错误判断——将”有图标的选项”视为正常选项，将”无图标的选项”视为异常选项。实际上恰恰相反：

选项 来源 实际状态
有图标 10_linux（传统方式） 蓝屏
无图标 15_uki（UKI 方式） 正常

禁用 15_uki 并删除 UKI 文件后，系统中仅保留 10_linux 生成的蓝屏启动项。重启后 GRUB 菜单只剩一个 Arch Linux 选项，选择后必然蓝屏，系统完全无法启动。

六、正确修复方案

6.1 修复思路

  • 禁用 10_linux，阻止它生成有问题的传统启动项
  • 在 40_custom 中手动编写正确的 chainloader 启动项
  • 保留 15_uki 作为备用（或禁用以避免重复条目）
  • 手动编写的启动项包含 --class arch，确保主题图标正确显示

6.2 修复步骤

第一步：禁用 10_linux 脚本

此操作不会删除文件，仅取消其可执行权限。grub-mkconfig 在执行时会跳过没有可执行权限的脚本。

第二步：编辑 40_custom 添加自定义启动项

在文件末尾（注释行之后）添加以下内容：

各字段说明：

字段 说明
--class arch 主题图标匹配标识，使 BigSur 主题显示 Arch 图标
insmod part_gpt 加载 GPT 分区表支持模块
insmod fat 加载 FAT 文件系统支持模块
insmod chain 加载 chainloader 支持模块
search ... BCD2-2326 通过 UUID 定位 EFI 分区
chainloader /EFI/Linux/arch-linux.efi 加载 UKI 文件

注意：BCD2-2326 为 EFI 分区的 UUID，需要替换为实际值。可通过 blkid /dev/nvme0n1p1 查看。

八、总结

项目 内容
问题根源 10_linux 脚本在 UKI 模式下生成了缺少 initramfs 的传统启动项
直接原因 10_linux 无法感知系统的 UKI 模式，按传统方式生成配置
